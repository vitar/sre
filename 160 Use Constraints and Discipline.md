# Use Constraints and Discipline

## Status

Draft v0.1

## Purpose

This document consolidates the practical constraints, usage rules, and interpretive discipline required for the model to be applied consistently.

Its purpose is not to add new theory. Its purpose is to make misuse visible.

The core model, classification protocol, metrics guidance, failure definition, and intervention logic already contain many constraints. At present, however, those constraints are distributed across documents. This makes disciplined use harder to enforce and easier to drift.

This document gathers the most important constraints into one place.

It is a companion to:

* **Core Model Definition v0.2**
* **Classification Protocol**
* **Metrics and Proxies**
* **Failure and Breakdown Definition**
* **Intervention Logic**

---

## 1. Why a Discipline Document Is Needed

A model can fail in practice even when its core structure is sound.

This usually happens not because the model is false, but because it is used without disciplined constraints.

Typical failure modes of use include:

* hidden boundary shifts,
* rhetorical contour labeling,
* forced classification,
* false precision,
* time-horizon collapse,
* mediator blindness,
* invalid comparison,
* and intervention without explicit trade-off.

If such usage remains unconstrained, the model becomes elastic enough to explain anything and precise enough to mislead.

This document exists to reduce that risk.

---

## 2. Core Discipline Principle

The model should be used as a **disciplined structural lens**, not as a free-form metaphor.

This means that every serious use of the model should be able to answer, explicitly:

* what system is being analyzed,
* at what level,
* over what time horizon,
* using what classification logic,
* supported by what evidence or proxies,
* under what confidence level,
* and with what acknowledged limitations.

If those conditions are not met, the use may still be exploratory, but it should not be treated as analytically strong.

## 3. Non-Negotiable Use Constraints

The following constraints should be treated as mandatory for disciplined use.

### 3.1 Boundary Declaration Constraint

No serious diagnosis, failure claim, intervention recommendation, or comparison should be made without declaring the system boundary.

At minimum, this includes:

* focal system,
* nesting level,
* main excluded externalities.

### 3.2 Time Horizon Constraint

No contour classification, failure judgment, or intervention recommendation should be made without stating the relevant time horizon.

At minimum:

* short-term,
* medium-term,
* or long-term.

### 3.3 Primary Intent Constraint

Contour classification must begin with primary functional intent, not with visible downstream consequences or rhetorical labels.

### 3.4 Mediator Constraint

Not every important object should be forced into Survival, Reproduction, or Evolution.

If the object primarily shapes allocation, signaling, coordination, legitimacy, or governance, it should first be treated as a mediator.

### 3.5 Trade-Off Constraint

Every serious intervention claim must state an expected trade-off.

No intervention should be treated as free of cost, displacement, delay, or sacrifice.

### 3.6 Confidence Constraint

Every proxy-supported estimate, failure claim, or intervention recommendation should state a confidence level.

At minimum:

* high,
* medium,
* or low.

### 3.7 Ambiguity Constraint

Ambiguity should be recorded explicitly rather than hidden by forced certainty.

Acceptable ambiguity labels include:

* mixed-primary,
* boundary-sensitive,
* time-sensitive,
* mediator-dominant,
* low-confidence.

### 3.8 Anti-Precision Constraint

Do not convert the model into exact numerical claims unless the boundary, proxy quality, and scoring logic are strong enough to justify it.

### 3.9 Comparison Constraint

No state comparison should be treated as valid unless the compared cases share sufficiently aligned:

* boundary logic,
* horizon logic,
* proxy logic,
* and contour interpretation discipline.

This document defines the constraint; detailed comparison rules belong to a separate comparability document.

## 4. Required Analysis Inputs

Before using the model in a non-trivial way, the following inputs should be declared.

### 4.1 System Boundary

What system is being analyzed?

### 4.2 Nesting Level

At what level is the system being analyzed?

Examples:

* subsystem,
* team,
* organization,
* market,
* state.

### 4.3 Time Horizon

What horizon is relevant to the claim?

### 4.4 Classification Object

What object is being classified, assessed, or acted on?

### 4.5 Evidence Basis

What supports the claim?

Examples:

* direct observation,
* proxy set,
* documented allocation,
* inferred pattern,
* case evidence,
* reasoned interpretation.

### 4.6 Confidence Level

How strong is the support for the claim?

These are the minimum conditions for disciplined application.

## 5. Rules for Diagnosis

### 5.1 Do Not Diagnose Without Boundary

A contour diagnosis without boundary declaration is incomplete.

### 5.2 Do Not Diagnose from Single Signals Alone

Avoid drawing a strong state estimate from a single indicator when multi-proxy support is available.

### 5.3 Distinguish Distortion from Failure

Do not escalate from imbalance to failure language without checking breakdown, degradation, compensation margin, and functional continuity.

### 5.4 Distinguish Formal Survival from Functional Survival

The persistence of visible structure is not sufficient evidence of viability.

### 5.5 Record Mediator Effects Separately

If a mediator is clearly shaping the state, note it rather than hiding it inside contour language.

---

## 6. Rules for Classification

### 6.1 Primary Intent First

Do not classify based only on visible output.

### 6.2 Horizon Must Be Explicit

A classification without horizon is incomplete.

### 6.3 Record Secondary Effects Separately

Do not let secondary consequences erase the primary contour assignment.

### 6.4 Use Mediator-First When Needed

Do not force governance, coordination, meaning, power, legitimacy, or trust structures into a contour if their primary function is cross-cutting.

### 6.5 State Uncertainty Explicitly

If classification is ambiguous, state why.

---

## 7. Rules for Proxy Use

### 7.1 Proxies Support Interpretation; They Do Not Replace It

The model still requires judgment.

### 7.2 Proxy Sets Are Better Than Single Proxies

Use grouped indicators where possible.

### 7.3 Proxy Meaning Depends on Boundary and Horizon

The same metric may indicate different contour emphasis under different conditions.

### 7.4 Declared Intent Is Not Enough

Budgets, labels, or official strategy statements are not automatically valid evidence of actual allocation.

### 7.5 Low-Quality Proxies Lower Claim Strength

Weak measurement basis should lower confidence, not be ignored.

---

## 8. Rules for Failure Claims

### 8.1 Failure Claims Must Be Narrow

Use the narrowest accurate failure formulation.

Examples:

* local breakdown,
* long-horizon Evolution failure,
* mediator-induced degradation,
* formal survival with functional identity failure.

### 8.2 Failure Requires More Than Imbalance

Do not treat contour distortion as failure by default.

### 8.3 Compensation Margin Must Be Checked

A system may remain viable through compensation, even under serious stress.

### 8.4 Collapse Should Be Reserved

Do not use collapse language unless recovery within the current form is no longer meaningfully plausible.

---

## 9. Rules for Intervention Claims

### 9.1 Diagnose Before Prescribing

No intervention recommendation should be treated as disciplined unless a diagnosed condition is stated.

### 9.2 Select the Narrowest Intervention Class That Fits

Do not jump to reconstitution when stabilization or mediator repair is sufficient.

### 9.3 Declare the Target of Intervention

State whether the move is directed at:

* Survival,
* Reproduction,
* Evolution,
* Mediator,
* or Boundary.

### 9.4 Declare the Trade-Off

Intervention claims without trade-offs are under-specified.

### 9.5 Declare the Reassessment Trigger

Every intervention should specify when it will be reviewed, stopped, or revised.

---

## 10. Invalid Uses

The following uses should be treated as invalid or analytically weak.

### 10.1 Free-Floating Metaphor Use

Using the contours loosely without declared boundary, horizon, or classification object.

### 10.2 Totalized Explanation

Claiming that the model fully explains all relevant complexity in a domain without acknowledging mediators, omitted variables, or scope limits.

### 10.3 Rhetorical Rebranding

Relabeling familiar opinions in contour language without adding analytical discipline.

### 10.4 Numeric Theater

Assigning exact numeric values to contour states without sufficient measurement basis.

### 10.5 Cross-Level Drift

Switching silently between individual, organizational, and societal levels inside the same argument.

### 10.6 Intervention by Intuition Alone

Using the model’s language to justify action without explicit diagnosis, trade-off, or reassessment logic.

### 10.7 Comparison Without Alignment

Comparing states across cases with incompatible boundaries, horizons, or proxy structures.

## 11. Allowed Use Modes

Not every use of the model needs the same level of rigor. The following modes are acceptable if labeled clearly.

### 11.1 Exploratory Use

Used for early thinking, rough sensemaking, or idea generation.

Requirements:

* boundary stated if possible,
* uncertainty admitted,
* no strong failure or intervention claims.

### 11.2 Diagnostic Use

Used to describe system state, contour posture, mediator effects, or distortion patterns.

Requirements:

* boundary declared,
* horizon declared,
* classification object stated,
* evidence basis stated,
* confidence stated.

### 11.3 Comparative Use

Used to compare cases, periods, or units.

Requirements:

* diagnostic discipline met,
* aligned comparison basis,
* comparison limits stated.

### 11.4 Intervention Use

Used to recommend action.

Requirements:

* diagnostic discipline met,
* intervention class stated,
* trade-off stated,
* reassessment trigger stated.

### 11.5 Formalization Use

Used to test assumptions, derive laws, refine metrics, or standardize artifacts.

Requirements:

* strict terminology,
* explicit scope,
* revision discipline.

## 12. Enforcement Heuristics

The following quick checks help identify misuse.

### 12.1 Boundary Check

Can the user state exactly what system is being analyzed?

### 12.2 Horizon Check

Can the user state whether the claim is short-, medium-, or long-term?

### 12.3 Classification Check

Can the user explain why the object belongs primarily to a given contour or mediator?

### 12.4 Evidence Check

Can the user point to at least one meaningful basis for the claim?

### 12.5 Trade-Off Check

Can the user state what is expected to worsen, shrink, slow, or be constrained by the proposed move?

If several of these checks fail, the use should be treated as exploratory at best.

## 14. Relationship to Other Documents

This document does not replace:

* contour definitions,
* classification rules,
* proxy guidance,
* failure definitions,
* or intervention logic.

It acts as an enforcement layer across them.

Its role is to make those documents easier to use together without hidden drift.

---

## 15. Working Summary

This document defines the discipline required to use the model without stretching it into vague metaphor or false precision.

Its central claim is practical:

**the model becomes more reliable when its users are forced to declare boundary, time horizon, classification logic, evidence basis, confidence, and trade-offs explicitly.**

Used this way, constraints are not a burden on the model. They are what make the model trustworthy.
